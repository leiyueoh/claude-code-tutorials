# 1. permission mode（权限控制模式）

==**以下四个模式（manual mode / accept edits mode / auto mode / plan mode）可以通过 shift + tab 按键组合进行切换**==

---

## manual mode（手动模式）

- ==**控制策略**：**零信任，逐项审批==**
    
- **行为表现**：这是最安全的防御性模式。Claude 只要尝试修改文件、创建新文件或在终端运行任何一条 Shell 命令，都会立即暂停，并在控制台弹出提示等待输入进行确认
    
- **适用场景**：涉及核心业务逻辑的修改、复杂的系统级命令调试，或者在你对 AI 的修改意图还不够确定、需要绝对掌控权时使用

==**这个是默认模式，启动 claude code 自动选择该模式**==

![[mode-1.png]]

---

## accept edits mode（接受编辑模式）

- ==**控制策略**：**自动修改文件，命令仍需审批**==
    
- **行为表现**：在这个模式下，你授予了 Claude Code 直接改动代码库中文件的权限。它在生成代码后会自动写入文件，无需你每次都手动点击同意。但是，如果它需要运行终端命令，系统依然会暂停并向你申请权限
    
- **适用场景**：进行大面积的代码重构、修复大批量的 Lint 错误或编写常规业务代码。开发效率大幅提升

![[mode-2.png]]

---

## auto mode（自动模式）

- ==**控制策略**：**完全信任，全自动执行**==
    
- **行为表现**：这是自动化程度最高的模式。Claude Code 拥有完整的操作权限，不仅可以自由修改和创建文件，还能自主决定并执行终端命令（例如在定位 Bug 时，它会自动运行测试、根据报错修改代码、再次运行测试，直到问题解决）。只有在遇到极度高危的命令或完成最终任务时才会提示你
    
- **适用场景**：自动化流水线任务、独立的 Bug 修复测试、或者在隔离的沙盒/开发容器（Dev Container）环境中进行快速迭代。由于其具备直接执行命令的权限，在生产环境或未提交的本地重要分支上建议谨慎使用

![[mode-3.png]]

---

## plan mode（计划/调研模式）

==- **控制策略**：**绝对只读，完全禁止修改**==
    
- **行为表现**：在这个模式下，Claude 被剥夺了所有写文件和执行实质性系统命令的权限。它只能读取代码库（Read-only）、搜索文件、分析代码结构并生成技术方案。如果任务需要修改代码，它会把完整的修改计划以文本形式呈现给你，但绝不会动手改动任何一个字节
    
- **适用场景**：大型项目的架构分析、复杂 Bug 的根因调研、在动手前的方案评审（Code Review / Design Review），或者你单纯想让 AI 帮你解释某段复杂代码的逻辑，确保绝对不会污染代码库

==**日常使用推荐这个**==

![[mode-4.png]]

---

| **权限模式**        | **文件编辑**     | **命令执行**       | **用户介入频率** | **效率与安全侧重**   | **典型适用场景**             |
| --------------- | ------------ | -------------- | ---------- | ------------- | ---------------------- |
| **Plan**        | **完全禁止**（只读） | **完全禁止**       | 极低         | 绝对安全，零风险      | 架构分析、Bug 根因调研、方案评审     |
| **Manual**      | 每次都需要手动审批    | 每次都需要手动审批      | 极高         | 极高安全性，效率较低    | 敏感代码重构、关键系统命令调试        |
| **acceptEdits** | **自动执行**     | 每次都需要手动审批      | 中等         | 兼顾文件编辑效率与环境安全 | 日常业务开发、大面积重构与 Lint 修复  |
| **Auto**        | **自动执行**     | **自动执行**（高危除外） | 极低         | 极高效率，依赖环境隔离   | 自动化流水线任务、沙盒环境下的 Bug 修复 |

# 2. slash command selection mode（斜杠指令选择模式）

==**以下三种命令是在交互终端内引导 AI 执行特定任务、定义工作流逻辑或拆解复杂目标的指令**==

---

## /plan

- **功能定义**：命令 Claude 针对你提出的复杂需求，先不要动手写代码，==而是输出一份详尽的**分步执行计划（Roadmap）**==，与用户确定好计划后，再一次性精准执行项目
    
- **行为表现**：它会分析当前代码库的相关文件，列出解决问题所需的步骤、可能受影响的文件模块以及潜在的风险点，等待你对计划进行评审和确认

 **plan mode 和 /plan 概念本质的区别：

- `Plan Mode`（权限模式）与 `/plan`（斜杠命令）在 Claude Code 中属于完全不同的维度，它们的根本区别在于：**一个是全局的“安全框架”，一个是具体的“动作指令”**
	
- **`Plan Mode` 是一种“状态（State）”**：它决定了 Claude **能做什么（权限范围）**。一旦开启，它就像一个全局的安全沙箱，只要你在这个状态下，Claude 绝对没有权限改动任何文件或执行系统命令，处于纯只读状态
    
- **`/plan` 是一种“行为（Action）”**：它决定了 Claude **怎么去做（工作流程）**。它是你主动下达的指令，意思是“请先为我拆解这个任务的步骤，不要盲目动手”

==其实日常使用两个没有太大区别==

![[mode-5.png]]

==**使用方式**：/plan + 空格 + 语言描述==

![[mode-6.png]]

**claude code 会给出几个选择题，跟用户进行确定任务详情**

**例（如上图所示）**：
- 1、2、3都是 claude code 所给出的方案
- 4为针对于当前问题，用户的方案（如果 claude code 的方案不合适）
- 5则为该问题不确定时，可跟 claude code 进行讨论，再做决定

## /loop（循环自动迭代命令）

- **功能定义**：开启一个“自动尝试 -> 运行测试 -> 报错修复 -> 再次测试”的闭环控制流，直到设定的条件达成或达到最大循环次数
    
- **行为表现**：当你给出一个长任务（如“修复所有单元测试错误”）时，`/loop` 会让 claude code 进入自动化调试状态。它会不断根据终端的报错信息自我修正代码，并自动重新运行命令，减少用户在中间环节频繁输入指令的干预
    
- **安全提示**：由于 `/loop` 带有自动化执行命令的属性，通常建议在受信任的项目或沙盒环境中使用，以防陷入死循环或产生非预期的系统级副作用

![[mode-7.jpg]]


==示例说明==：

```
/loop 5m 检查我的 PR，处理 review 评论，修复失败的 CI
```

## /goal（设定核心目标命令）

- **功能定义**：为当前会话注入或重置一个**最高优先级的全局目标（KPI）**
    
- **行为表现**：在长对话中，AI 很容易随着技术细节的深入而偏离最初的方向。通过 `/goal`，你可以显式地锚定“这次对话的终极任务是什么”。此后，Claude 在执行任何子任务或给出品控建议时，都会强制围绕这个核心目标进行对齐，防止上下文漂移

![[mode-8.jpg]]

![[mode-9.jpg]]

==示例说明==：

```
用 /goal 命令定义成功标准，claude code 会反复迭代，直到达标或者达到你设的轮次上限

比如：

/goal 把首页 Lighthouse 分数提到 90 以上，最多试 5 次

关键区别在哪？在回合制里，claude code 每做完一步就会停下来等你。因为它不确定什么算"够好"，所以倾向于提前交差

/goal 解决的就是这个问题：你把"什么算完成"说清楚了，claude code 就不用猜了。每次它试图停下来，一个评估模型会检查目标是否达成，没达成就打回去继续
```
