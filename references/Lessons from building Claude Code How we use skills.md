技能已成为Claude Code中最常用的扩展功能之一。它们灵活、易于创建和分发。

但这种灵活性也使得我们难以确定哪种方法最有效。哪些技能值得培养？如何构建技能体系？何时与他人分享这些技能？

在 Anthropic，我们广泛使用 Claude Code 中的各种技能，其中数百项技能都已投入使用。以下是我们从运用这些技能加速产品开发过程中总结出的经验教训。

## 什么是 skill ？

技能是包含指令、脚本和资源的文件夹，智能体可以发现并使用它们来更准确、更高效地完成任务。本文假设您已熟悉技能的基础知识；如果您是新手，请先学习 [Skilljar 上的智能体技能入门课程](https://anthropic.skilljar.com/introduction-to-agent-skills) 。

我们经常听到的一种关于技能的误解是，它们“只是 Markdown 文件”。实际上，它们是文件夹，可以包含脚本、资源、数据等，代理可以发现、探索和操作这些资源。

在 Claude Code 中，技能还具有 [多种配置选项，](https://code.claude.com/docs/en/skills#frontmatter-reference) 包括注册动态钩子。

我们发现 Claude Code 中一些最有效的技能有效地利用了这些配置选项和文件夹结构。

## skill 类型

在对Anthropic公司内部所有技能进行分类后，我们发现它们可以归纳为九大类。最优秀的技能可以清晰地归入某一类；而那些试图面面俱到的技能则往往跨越多个类别，令员工感到困惑。这并非一份详尽的清单，但它为识别自身技能库中的不足之处提供了一个有用的框架。

![Claude Code 团队对我们的内部技能进行了分类，发现它们可以归纳为九个不同的类别。](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a1f3a763cec27e2f026439c_b7942952.png)

Claude Code 团队对我们的内部技能进行了分类，发现它们可以归纳为九个不同的类别。

### 1\. 库和 API 参考

这些技能讲解了如何正确使用库、命令行界面 (CLI) 或软件开发工具包 (SDK)。它们既包括内部库，也包括 Claude Code 有时难以处理的常用库。这些技能通常包含一个参考代码片段文件夹，以及一份 Claude 在编写脚本时需要避免的陷阱列表。

例如：

- `billing-lib` — 您的内部计费库：特殊情况、意外情况等。
- `internal-platform-cli ` — 内部 CLI 包装器的每个子命令，以及何时使用它们的示例。
- `sandbox-proxy ` — 为开发工作配置组织的出口网关：哪些主机可达，如何调试“连接被拒绝”错误，如何添加允许列表条目。

### 2\. 产品验证

这些技能描述了如何测试或验证代码是否正常运行。它们通常与 Playwright、tmux 或其他外部验证工具配合使用。

验证技能对克劳德公司内部的输出质量产生了最显著的影响。让工程师花一周时间专门提升验证技能是值得的。

可以考虑一些技巧，例如让 Claude 录制输出视频，以便您准确了解它测试的内容；或者在每个步骤中强制执行状态的程序化断言。这些通常是通过在技能中包含各种脚本来实现的。

例如：

- `signup-flow-driver ` — 在无头浏览器中运行注册 → 电子邮件验证 → 新手引导流程，并在每个步骤中提供状态断言钩子
- `checkout-verifier ` — 使用 Stripe 测试卡驱动结账用户界面，验证发票是否实际进入正确的状态
- `tmux-cli-driver ` — 用于交互式 CLI 测试，其中您要验证的内容需要 TTY。

### 3\. 数据获取与分析

这些技能与您的数据和监控系统密切相关。这些技能可能包括使用凭据、特定仪表板 ID 等获取数据的库，以及有关常见工作流程或数据获取方法的说明。

例如：

- `funnel-query ` —“我应该加入哪些事件才能查看注册→激活→付费”以及包含规范用户 ID 的表
- `cohort-compare` — 比较两个用户群的留存率或转化率，标记具有统计学意义的差异，并链接到细分定义
- `grafana ` — 数据源 UID、集群名称、问题 → 仪表板查找表
- `datadog ` — 字段引用（@request\_id 与 trace\_id）、服务列表、指标前缀约定

### 4\. 业务流程和团队自动化

这些技能可以将重复性工作流程自动化为一条命令。这些技能通常是相当简单的指令，但可能与其他技能或主控程序 (MCP) 存在更复杂的依赖关系。对于这些技能，将先前的结果保存到日志文件中有助于模型保持一致性，并反映工作流程的先前执行情况。

例如：

- `standup-post ` — 汇总您的工单跟踪器、GitHub 活动和之前的 Slack 记录 → 格式化的站会，仅显示增量数据
- `create-<ticket-system>-ticket` — 强制执行架构（有效的枚举值、必填字段）以及创建后工作流程（通知审核人员、在 Slack 中提供链接）
- `weekly-recap` — 合并的 PR + 已关闭的工单 + 已部署 → 格式化的总结帖子

### 5\. 代码脚手架和模板

这些技能可以为代码库中的特定功能生成框架样板。您可以将这些技能与可组合的脚本结合使用。当您的脚手架需要自然语言支持，而这些需求无法完全通过代码实现时，它们尤其有用。

例如：

- `new-<framework>-workflow ` — 使用您的注解搭建新的服务/工作流/处理程序
- `new-migration ` — 您的迁移文件模板以及常见陷阱
- `create-app` — 全新的内部应用程序，已预先配置好您的身份验证、日志记录和部署功能

### 6\. 代码质量和审查

这些技能可以确保组织内部的代码质量，并有助于代码审查。它们可以包括用于最大限度提高健壮性的确定性脚本或工具。您可能希望将这些技能作为钩子的一部分或在 GitHub Action 中自动运行。

- `adversarial-review ` — 生成一个具有全新视角的子代理进行批判，实施修复，不断迭代，直到发现的问题都变成了吹毛求疵。
- `code-style ` — 强制执行代码风格，特别是 Claude 默认情况下不太擅长的风格。
- `testing-practices` — 关于如何编写测试以及测试什么内容的说明。

### 7\. CI/CD 和部署

这些技能可以帮助你在代码库中获取、推送和部署代码。这些技能可能还会引用其他技能来收集数据。

例如：

- `babysit-pr ` — 监控 PR → 重试不稳定的 CI → 解决合并冲突 → 启用自动合并
- `deploy-<service>` — 构建 → 冒烟测试 → 逐步上线并比较错误率 → 回归测试自动回滚
- `cherry-pick-prod` — 隔离工作树 → 选择性提取 → 冲突解决 → 使用模板进行 PR

### 8\. 运行手册

这些技能包括：获取症状（例如 Slack 线程、警报或错误签名），进行多工具调查，并生成结构化报告。

例如：

- `<service>-debugging ` — 绘制症状图 → 工具 → 查询模式，找到您流量最高的服务
- `oncall-runner ` — 获取警报 → 检查常见嫌疑对象 → 格式化调查结果
- `log-correlator ` —给定一个请求 ID，从所有可能访问过该请求的系统中提取匹配的日志。

### 9\. 基础设施运营

这些技能涵盖日常维护和操作流程，其中一些操作可能具有破坏性，因此需要采取防护措施。这些措施有助于工程师在关键操作中遵循最佳实践。

例如：

- `<resource>-orphans` — 查找孤立的 pod/卷 → 发布到 Slack → 等待一段时间 → 用户确认 → 级联清理
- `dependency-management ` — 贵组织的依赖项审批工作流程
- `cost-investigation ` — “为什么我们的存储/出站流量费用飙升？”并提供具体的存储桶和查询模式

## 提升技能的技巧

确定要开发的技能后，该如何编写代码呢？以下是 Claude Code 团队在开发技能方面总结的一些最佳实践、技巧和窍门。

### 不要说显而易见的事

![](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a1f3a763cec27e2f02643a2_6f109d87.png)

SKILL.md 文件指向 Claude 可以针对特定情况引用的其他几个文件。例如，如果某个任务处于待处理状态，它应该引用 stuck-jobs.md 文件。

Claude 已经懂得如何编程，也能阅读你的代码库。如果你发布的技能只是复述 Claude 的默认操作，那它只是提供了背景信息，并没有增加实际价值。如果你发布的技能主要涉及知识，那就应该专注于那些能够启发 Claude 跳出固有思维模式的信息。

前端 [设计技能](https://github.com/anthropics/skills/blob/main/skills/frontend-design/SKILL.md) 就是一个很好的例子；它是由 Anthropic 的一位工程师通过与客户反复沟通，改进 Claude 的设计品味而构建的，避免了 Inter 字体和紫色渐变等经典模式。

### 建立陷阱部分

任何技能中最具价值的内容都是“陷阱”部分。这些部分应该基于克劳德在使用技能时常遇到的失败点构建而成。理想情况下，你应该随着时间的推移不断更新技能，以涵盖这些陷阱。

例如：

“该 `subscriptions` 表仅支持追加更新。您需要的是版本号最高的行，而不是最新的行 `created_at` 。” “此字段 `@request_id ` 在 API 网关和 `trace_id` 计费服务中都会被调用，它们的值相同。” “即使 Stripe Webhook 实际上并未处理，暂存环境也会返回 200。请检查 `payment_events ` 实际状态。”

### 使用文件系统和渐进式披露

![](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a1f3a763cec27e2f026439f_0e0f23c0.png)

SKILL.md 文件指向 Claude 可以针对特定情况引用的其他几个文件。例如，如果某个任务处于待处理状态，它应该引用 stuck-jobs.md 文件。

正如我们之前所说，技能是一个文件夹，而不仅仅是一个 Markdown 文件。你应该把整个文件系统看作是一种上下文工程和渐进式披露。告诉 Claude 你的技能里有哪些文件，它会在适当的时候读取它们。

最简单的渐进式披露方式是指向其他 Markdown 文件供 Claude 使用。例如，您可以将详细的函数签名和用法示例拆分成多个文件 `references/api.md` 。

另一个例子：如果你的最终输出是 Markdown 文件，你可以包含一个模板文件供用户 `assets/` 复制和使用。

您可以创建文件夹，存放参考资料、脚本、示例等，这有助于 Claude 更有效地工作。

克劳德通常会尽量遵循你的指令，但由于技能可以重复使用，所以你的指令要谨慎，不要过于具体。给克劳德提供所需的信息，但也要赋予它适应不同情况的灵活性。

### 请为模型撰写描述，而不是为人类撰写描述。

当 Claude Code 启动会话时，它会创建一个包含所有可用技能及其描述的列表。Claude 会扫描此列表来判断“是否存在满足此请求的技能？”这意味着描述字段并非摘要，而是对何时触发此技能的详细说明。

在技能描述中加入“照看孩子”之类的触发词会很有帮助。

### 帮助克劳德记住

![](https://cdn.prod.website-files.com/68a44d4040f98a4adf2207b6/6a1f3a763cec27e2f02643b1_9159a9b1.png)

这个文本日志文件帮助 Claude 记住过去发生的事件，例如审查 Sarah 的授权 PR。

某些技能可以通过存储数据来实现某种形式的记忆。你可以将数据存储在任何类型的存储介质中，例如仅追加的文本日志文件或 JSON 文件，也可以存储到像 SQLite 数据库这样复杂的介质中。

例如，一项 `standup-post` 技能可能会保存一个 standups.log 文件，其中包含它编写的每篇文章，这意味着下次运行它时，Claude 会读取它自己的历史记录，并知道自昨天以来发生了哪些变化。

您可以使用环境变量 `${CLAUDE_PLUGIN_DATA} ` 来获取一个稳定的目录，您可以在其中存储数据，请在此处阅读更多关于持久化数据的技巧： [https://code.claude.com/docs/en/plugins-reference#persistent-data-directory](https://code.claude.com/docs/en/plugins-reference#persistent-data-directory) 。

### 存储脚本并生成代码

你能给 Claude 的最强大的工具之一就是代码。给 Claude 提供脚本和库，就能让 Claude 将精力集中在组合代码上，决定下一步该做什么，而不是重复编写样板代码。

例如，你的 `data-science` 技能库中可能包含一个函数库，用于从事件源获取数据。为了让 Claude 进行复杂分析，你可以给它提供一组辅助函数，如下所示：

### 使用按需钩子

技能可以包含仅在技能被调用时激活、且仅在当前游戏会话期间持续存在的钩子。这适用于一些你不想一直使用，但有时又非常有用的特殊钩子。

例如：

- `/` **`careful`** — 通过 Bash 上的 PreToolUse 匹配器阻止 rm -rf、DROP TABLE、force-push 和 kubectl delete 命令。只有在确定要操作生产环境时才需要启用此功能——始终启用会让人抓狂。
- `/` **`freeze`** — 阻止对指定目录之外的任何编辑/写入操作。在调试过程中非常有用：“我想添加日志，但我总是不小心‘修复’了无关的代码。”

## 技能分配

技能最大的好处之一就是可以与团队其他成员分享。

你可能希望通过两种方式与他人分享技能：

- 将你的技能提交到你的代码仓库（位于`./.claude/skills` ）
- 创建一个 **插件** ，并建立一个 Claude Code 插件市场，用户可以在其中上传和安装插件（更多 [文档信息](https://code.claude.com/docs/en/plugin-marketplaces) 请点击此处）。

对于规模较小的团队来说，将技能提交到代码库中是一个不错的选择。但每个提交的技能都会为模型增添一些上下文信息。随着团队规模的扩大，内部插件市场可以让你分发技能，让团队决定安装哪些技能，并包含安装流程。

## 管理技能市场

如何决定哪些技能可以放到技能交易平台上？人们如何提交技能信息？

在 Anthropic，我们没有一个集中式的团队来做决定；相反，我们努力自然而然地发现最有用的技能。如果有人想让其他人尝试某项技能，他们可以将其上传到 GitHub 的沙盒文件夹中，并在 Slack 或其他论坛上分享给其他人。

一旦某项技能获得关注（这取决于技能所有者的决定），他们就可以提交 PR 将其推向市场。

## 作曲技巧

您可能需要一些相互依赖的技能。例如，您可以创建一个文件上传技能来上传文件，以及一个 CSV 生成技能来生成并上传 CSV 文件。目前，市场或技能库尚未原生支持这种依赖关系管理，但您可以直接通过名称引用其他技能，如果这些技能已安装，模型会自动调用它们。

## 衡量技能

为了解某项技能的使用情况，我们使用 PreToolUse 钩子来记录公司内部的技能使用情况（ [示例代码在此](https://gist.github.com/ThariqS/24defad423d701746e23dc19aace4de5) ）。这意味着我们可以找到哪些技能受欢迎，哪些技能的使用率低于预期。

## 开始使用

技能最佳实践仍在不断发展。我们大多数优秀技能最初都只是寥寥几行代码和一个小小的陷阱，后来随着 Claude 遇到新的极端情况，大家不断完善这些技能，使其日臻完善。

了解技能的最佳方法是开始实践，不断尝试，看看哪些方法对你有效。
